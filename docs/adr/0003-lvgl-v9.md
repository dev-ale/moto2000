# ADR 0003: LVGL v9 as the UI library

Date: 2026-04-09
Status: Accepted

## Context

The ESP32 renders every dashboard screen on a 466×466 round AMOLED. We need smooth
animations, efficient partial redraws, round-display primitives, and a host simulator
for CI snapshot tests.

## Decision

Use **LVGL v9.2** as the UI library, as an ESP-IDF managed component. Use the
LVGL SDL host simulator for CI snapshot tests.

## Consequences

Positive:

- Mature, well-documented, actively maintained
- First-class round-display support (arcs, circular scales, rotation)
- SDL host simulator enables snapshot tests on Linux CI without hardware
- Efficient partial redraws suit AMOLED power characteristics
- Large widget library (gauges, graphs, labels) covers most PRD screens

Negative:

- Memory footprint is non-trivial; must tune buffers for the 466×466 display
- v9 is a breaking change from v8; community examples sometimes lag

Rejected alternatives:

- **Custom blitter** — too much work for no benefit
- **TFT_eSPI / Arduino graphics** — not available under ESP-IDF without porting;
  weaker widget library

## Amendment — 2026-04-09: host simulator diverges from on-device stack

Slice 1.5b (PR #29) shipped the host simulator at
`hardware/firmware/host-sim/` with a **pure-C software rasteriser and an
8×8 bitmap font**, not LVGL + SDL as this ADR originally anticipated.
The on-device stack (Slice 2 onwards) still uses LVGL v9.2 as decided
above; this amendment records the split and the reasoning.

### Why the host simulator does not use LVGL

- **CI snapshot determinism.** The snapshot tests diff PNGs byte-for-byte.
  Real text shapers (freetype + LVGL font rendering) produce subtly
  different output across macOS / Linux / CI runner versions, which
  would make the tests flaky without heroic normalisation work.
- **Build complexity.** Bringing LVGL + SDL2 + freetype + fontconfig into
  a host project that has to work on both macOS and `ubuntu-latest` is
  substantial. The pure-C renderer uses only two single-header stb libs.
- **Clean seam.** The renderer sits behind the `host_sim_render_*`
  functions in `host-sim/include/host_sim/renderer.h`. A future slice can
  replace the backend with real LVGL (or add it as an optional configure
  flag) without touching the PNG writer, the snapshot harness, the
  CMake wiring, or the Swift `HostSimulatorBLETransport` that pipes
  bytes in.

### What this means for visual expectations

**The host simulator PNGs are NOT representative of the final on-device
look.** The real ESP32 runs LVGL v9 with real TTF fonts at ~267 PPI on a
466×466 AMOLED — comparable to an iPhone 4 Retina screen — and will
render anti-aliased text, smooth curves, and proper typography matching
the mockups in `docs/mockups.html`. The simulator deliberately looks
retro to keep CI tests deterministic.

When reviewing snapshot diffs, judge **layout, information density, and
glanceability**, not typographic polish. Typographic polish is the real
firmware's job.

### Open follow-up

If a future screen slice needs rich typography or vector graphics that
outgrow the 8×8 bitmap font, add an **optional** LVGL backend to the
host simulator, selected at configure time so CI can keep running the
lightweight path. Do not replace the bitmap renderer — the two serve
different purposes (regression testing vs. design preview).
